# Persistent config and job state via `~/.config/scribd-dl/`

**Date:** 2026-06-10
**Scope tier:** Standard
**Status:** Ready for planning

## Summary

Engine получает персистентное состояние через два файла в `~/.config/scribd-dl/`: `settings.json` хранит пользовательскую настройку `outputFolder`, `jobs.jsonl` — state-snapshot очереди (одна строка на job). На старте engine читает оба файла, превращает все `Downloading` в `Queued` и продолжает работу. CLI-режим `bun start <url>` удаляется целиком — единственная точка запуска движка теперь engine sidecar, остальные процессы (TUI, SPA, будущий Tauri-desktop) — клиенты поверх его HTTP/WS.

## Problem Frame

Сейчас состояние engine живёт только в памяти: `Ref<Map<JobId, Job>>` и `PubSub<JobEvent>` в `packages/engine/src/service/DownloadEngine.ts`. При рестарте процесса теряется и очередь, и пользовательский `outputFolder` (он мутируется в runtime через WS-эвент `OutputFolderChanged`, но никуда не пишется). Все три клиента — TUI, SPA, будущий desktop — стартуют с пустым snapshot независимо от того, что у пользователя было в прошлой сессии.

Параллельно CLI-флаги (`--output`, `--filename`, `--rendertime`) — единственный способ задать конфиг, что не сочетается с UI, который сам хочет менять папку. И они никуда не сохраняются: один запуск = один набор флагов.

Перенос состояния в файлы решает обе проблемы сразу: сессия переживает рестарт, UI-изменения папки видны при следующем запуске, а CLI-флаги для конфигурации становятся не нужны.

## Requirements

### R1. Файловое расположение

- R1.1. Каталог хранения: `~/.config/scribd-dl/` на всех платформах, путь строится через `path.join(os.homedir(), ".config", "scribd-dl")`. Каталог создаётся при первой записи если его нет.
- R1.2. Имена файлов: `settings.json`, `jobs.jsonl`.
- R1.3. Если каталог или файлы отсутствуют — engine стартует с дефолтами без ошибки.
- R1.4. Если файл повреждён (невалидный JSON / неполная строка JSONL) — engine логирует warning, использует дефолты (для settings) или игнорирует повреждённую строку (для jobs.jsonl), не падает.

### R2. `settings.json`

- R2.1. Формат: один JSON-объект.
- R2.2. Поля: только `outputFolder: string` (абсолютный путь, уже expanded, без `~`).
- R2.3. Дефолт `outputFolder` при отсутствии файла остаётся таким же, как сейчас в `DEFAULT_CONFIG` (`packages/engine/src/utils/io/ConfigLoader.ts`).
- R2.4. Файл перезаписывается атомарно при любом изменении `outputFolder` (через HTTP `POST /folder` или эквивалентный механизм).
- R2.5. Перезапись engine'ом — единственный писатель. Внешние редакторы поддерживать не требуется.

### R3. `jobs.jsonl`

- R3.1. Формат: JSON Lines, одна строка = один `Job` (текущая форма из `packages/shared/src/jobs.ts`).
- R3.2. Файл представляет state-snapshot текущей очереди, а не event log. История прогресса не сохраняется.
- R3.3. Файл перезаписывается атомарно (`writeFile(tmp) + rename`) при каждой смене статуса любого job (`Queued → Downloading → Downloaded | Failed`, `add`, `remove`, `retry`, `requeue`).
- R3.4. Прогресс-события (`JobProgress`, `JobTitleUpdated`) в файл не пишутся — они только бродкастятся по WS. Файл перезаписывается только когда меняется то, что переживает рестарт.
- R3.5. При старте engine сериализует строки в `Map<JobId, Job>`, восстанавливая snapshot до начала работы воркера.

### R4. Семантика рестарта

- R4.1. Все job'ы со статусом `Downloading` на момент чтения файла молча переводятся в `Queued` (без подтверждения пользователя) и попадают в воркер-очередь по порядку.
- R4.2. Job'ы со статусом `Queued` остаются `Queued` и попадают в воркер-очередь.
- R4.3. Job'ы со статусом `Downloaded` и `Failed` остаются как есть и видны клиентам в snapshot.
- R4.4. Воркер начинает работу с первого `Queued` в порядке, в котором они оказались в Map.

### R5. Удаление CLI-режима

- R5.1. `packages/engine/run.ts` удаляется. `bun start` либо переводится на запуск engine (`bun run engine` эквивалент), либо убирается из `package.json`.
- R5.2. CLI-флаги `--output`, `--filename`, `--rendertime` удаляются.
- R5.3. `filename`-стратегия (`title` vs `id`) и `rendertime` фиксируются константами в коде (`DEFAULT_CONFIG`) без возможности переопределения извне.
- R5.4. `outputFolder` задаётся только через `settings.json` (и через HTTP `POST /folder` от клиентов). CLI-способа задать папку больше нет.
- R5.5. Логика парсинга URL из текста (`extractUrls` в `DownloadEngine.ts`) остаётся — она нужна для клиентов, принимающих многострочный ввод.

### R6. Очистка истории

- R6.1. Добавляется HTTP endpoint `DELETE /jobs/completed` — удаляет все job'ы со статусом `Downloaded`.
- R6.2. Добавляется HTTP endpoint `DELETE /jobs/failed` — удаляет все job'ы со статусом `Failed`.
- R6.3. Обе операции публикуют `JobRemoved` для каждого удалённого job'а и триггерят перезапись `jobs.jsonl`.
- R6.4. Контракт `DELETE /jobs/:id` (single remove) расширяется: разрешается удалять job'ы в любом статусе кроме `Downloading`. Сейчас `NotRemovable` возвращается для всех нечнэ-`Queued` — это правило снимается для `Downloaded` и `Failed`, остаётся для `Downloading`.

### R7. Клиентский контракт

- R7.1. HTTP `GET /snapshot` и WS-протокол не меняются. Клиенты по-прежнему получают `EngineSnapshot` на старте и поток `JobEvent` поверх.
- R7.2. Файлы `~/.config/scribd-dl/*` — внутренний контракт engine, клиенты не читают их напрямую.
- R7.3. Wire-типы в `packages/shared/src/jobs.ts` не меняются. Добавляются только новые HTTP request/response типы для clear-операций в `packages/shared/src/http.ts`.

## Key Decisions

### KD1. State-snapshot, не event log

`jobs.jsonl` хранит текущее состояние, а не append-only лог событий. Альтернатива (event log + replay при старте) дала бы полную историю и единый формат wire/file, но требует периодической компакции и replay-стоимость при каждом старте. Для self-use на десятках job'ов это лишняя сложность; state-snapshot решает основную задачу (пережить рестарт) самым прямым способом.

### KD2. JSONL, не SQLite/lowdb

Альтернатива через `bun:sqlite` (встроена, синхронная, транзакционная) — оверкилл для этого размера, плюс ломает плюс human-readable формата (можно посмотреть глазами, парсится чем угодно). JSONL даёт всё нужное в ~50 строк кода.

### KD3. Удаление CLI-режима целиком

Альтернативы — оставить CLI с file-конкуренцией (два процесса пишут в один файл) или сделать его read-only — обе создают сценарии, где состояние не сходится. Удаление одного из двух entry points полностью убирает класс багов «один писатель vs другой писатель» и сильно упрощает доменную модель: persistence — внутреннее дело engine, всё снаружи проходит через HTTP/WS.

### KD4. Auto-resume бывших Downloading без подтверждения

При рестарте `Downloading` молча переводятся в `Queued` и перескачиваются. Альтернатива (переводить в `Failed (retryable: true)` и ждать ручного retry) была бы аккуратнее в сценарии «engine упал из-за upstream-проблемы Scribd, после рестарта надо подумать, прежде чем пытаться снова», но для self-use потеря автоматизма перевешивает. Если в практике обнаружится частый кейс залипания на сетевых ошибках — пересмотреть.

### KD5. WS остаётся живым каналом независимо от файлов

`jobs.jsonl` перезаписывается только на смену статуса; progress-тики идут только по WS. Это разделяет «что переживает рестарт» (статус) от «что показывает прямо сейчас» (progress %). Один файл-snapshot был бы либо неточным (последний прогресс не виден после рестарта), либо дорогим (перезапись на каждый тик из десятков в секунду).

### KD6. `settings.json` — только `outputFolder`

`filename` и `rendertime` уезжают в код-дефолты. Это явный выбор: единственная настройка, которую пользователь реально хочет менять, — папка. Раздувать settings на «всё, что было CLI-флагом» — поощряет несуществующую сейчас потребность.

### KD7. Раздельные Clear completed / Clear failed

Завершённые успешно и упавшие — разные классы информации. Failed пользователь скорее всего хочет видеть дольше (что-то пошло не так, может вернётся retry-ить). Completed — почистил и забыл. Объединённый Clear-all потерял бы это различие, отдельные эндпоинты — нет.

## Scope Boundaries

- Отмена in-flight `Downloading` job (cancel). Сейчас single-fiber воркер не поддерживает отмену; реализация требует механизма interrupt и UX-решений вокруг. Отдельная фича, не в этом скоупе.
- Версионирование схемы `settings.json` и `jobs.jsonl` (поле `version`, миграции). Преждевременная защита для случая, который ещё не наступил. Добавляем при первой реальной миграции.
- File-based DB (SQLite, lowdb, level). Не нужен на этом размере данных (см. KD2).
- Конфигурация порта engine через файл. Остаётся CLI/env-аргументом `engine.ts`.
- Поддержка нескольких параллельных engine-процессов с координацией доступа к файлам. Один engine на машину; повторный запуск падает на `EADDRINUSE` для 4747 — этой защиты достаточно.
- Внешнее редактирование `settings.json` / `jobs.jsonl` пока engine запущен. Не поддерживаем — engine единственный писатель, внешние правки могут быть перетёрты.
- Логирование событий для аудита (отдельный event-log файл). Не нужен сейчас.

## Dependencies / Assumptions

- Engine процесс — единственный писатель в `~/.config/scribd-dl/`. Это держится за счёт удаления CLI-режима и привязки engine к фиксированному порту.
- `Bun.write` + `fs.promises.rename` дают атомарную замену файла на macOS/Linux, на которых работает проект.
- Текущий `JobEvent` контракт (`packages/shared/src/jobs.ts`) — достаточно полный, чтобы покрыть persistence + новые clear-операции через `JobRemoved`. Никаких новых событий не нужно.
- HTTP-слой engine (где живёт `POST /folder`, `DELETE /jobs/:id`) — место для добавления `DELETE /jobs/completed` и `DELETE /jobs/failed`. Конкретная файловая локация — задача планирования.
- Существующий `extractUrls` в `DownloadEngine.enqueue` остаётся источником истины для парсинга multi-URL ввода; клиенты передают сырой текст.

## Outstanding Questions

Нет — все развилки разрешены в диалоге.

## Success Criteria

- После kill -9 engine и рестарта: пользователь видит ту же папку и ту же очередь, что и до рестарта; все ранее `Downloading` job'ы перезапускаются автоматически.
- TUI и SPA, подключённые одновременно, видят одинаковый snapshot после рестарта engine.
- `bun start <url>` либо запускает engine, либо не существует — нет третьего пути со своим состоянием.
- HTTP API даёт явные операции `DELETE /jobs/completed` и `DELETE /jobs/failed`; клиенты могут построить UI-кнопки без дополнительной серверной логики.
- Удаление файлов в `~/.config/scribd-dl/` и старт engine = чистое состояние с дефолтами без падений.
