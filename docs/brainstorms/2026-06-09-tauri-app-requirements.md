# Tauri-обёртка для scribd-dl: requirements

**Дата:** 2026-06-09
**Статус:** Brainstorm → готово к `/ce-plan`
**Связано:** `run.js`, `src/App.js`, `src/utils/request/PuppeteerSg.js`, `config.ini`

## Проблема и цель

Сегодня скачивание = терминальная команда `bun start <url>`. Для разового использования это нормально, для повседневного — трение: открыть терминал, вспомнить команду, вставить URL. Цель — десктопное приложение, которое сводит весь UX к двум жестам: «открыл окно → `Cmd+V`».

**Метрика успеха:** скачать документ от запуска приложения до файла на диске за ≤ 2 действия пользователя (запустить + paste), без касания терминала или CLI-флагов.

## Пользователь и сценарий

Один пользователь (автор репо) на macOS. Дальше — возможно Windows/Linux. Сценарий: наткнулся на Scribd/Slideshare/Everand ссылку → запустил приложение из Spotlight/Dock → `Cmd+V` в окне → файл оказался в выбранной папке. Иногда — серия из нескольких ссылок подряд.

## Объём (in scope)

**Trigger model**
- Приложение запускается явно (Dock/Spotlight). Никаких background-tray, global hotkey, авто-watch клипборда.
- Внутри окна `Cmd+V`:
  - Если в буфере **поддерживаемый URL** (scribd / slideshare / everand) → задача сразу добавляется в очередь и стартует (если очередь пуста).
  - Иначе → нативный alert «Не похоже на поддерживаемую ссылку».

**Очередь и UI**
- Окно — список задач. Колонки: домен, заголовок (если уже известен) / URL, статус, прогресс.
- Статусы: `queued` → `downloading` → `done` | `failed`.
- Скачивания идут **последовательно** (одна за раз). Paste во время активной задачи → добавляется в `queued`.
- Без персистентной истории между сессиями: при закрытии приложения список очищается.

**Папка вывода**
- Дефолт: `~/Downloads`.
- В шапке окна виден текущий путь + кнопка смены → системный folder picker.
- Выбор персистится между сессиями (Tauri app data store).
- Применяется ко всем последующим скачиваниям. Изменение в середине очереди затрагивает ещё-не-стартовавшие задачи.

**Имя файла**
- Поведение из `config.ini` мигрируется как есть: при `[DIRECTORY] filename=title` → имя = заголовок документа; иначе → ID.
- На MVP это не настраивается через UI — фиксируем поведение `filename=title` как дефолт.
- **Коллизия имён:** если в папке уже есть файл с таким же именем, не относящийся к текущей очереди (т.е. с прошлых запусков или внешний) — добавляем суффикс `(1)`, `(2)`, … до первого свободного.

**Обработка ошибок**
- При фейле задачи → статус `failed`, видна короткая причина (последняя строка stderr / тип ошибки).
- В строке `failed` появляется кнопка **Retry** → задача возвращается в `queued` тем же URL.

**Защита от случайного закрытия**
- При попытке закрыть окно / выйти из приложения, если очередь не пуста или есть `downloading`-задача → confirm dialog: «Идёт скачивание / задачи в очереди. Закрыть?». Cancel → остаёмся.

**Платформа**
- MVP: macOS. .app bundle. Code signing / notarization — отложено (см. Outstanding).
- Код пишем cross-platform-ready (без macOS-only API в Rust части), но билд и тест только под mac.

## Объём (out of scope)

- **Global hotkey** и tray-app: триггер только внутри открытого окна.
- **Авто-watch клипборда** (privacy + навязчивость).
- **Параллельные скачивания**: только последовательно.
- **Персистентная история** между сессиями.
- **UI для filename strategy**: поведение зафиксировано на `title`.
- **Управление `config.ini`** через UI.
- **Windows / Linux билды** (только runtime-совместимость).
- **Code signing / notarization / автообновления.**

## Архитектура (нарочно фиксируем — это технический брейншторм)

**Sidecar-подход для MVP:**
- Существующий downloader (`src/App.js`, `src/service/*Downloader.js`, `PuppeteerSg`) **не переписывается**.
- Tauri app содержит Bun-рантайм как [external binary / sidecar](https://v2.tauri.app/develop/sidecar/).
- Rust часть спавнит `bun run.js <url>` как child process, читает stdout/stderr построчно, парсит прогресс и статус.
- Frontend (web tech внутри Tauri webview) рендерит очередь и подписан на события из Rust.

**Будущий трек (после MVP, отдельным брейнштормом):**
- Перенос downloader-логики на Rust + `chromiumoxide` / `headless_chrome` + `lopdf`/`printpdf`. Цель — отказаться от Bun-сайдкара и Chromium-бандла Puppeteer, ради меньшего размера приложения и одного языка. Это полный rewrite трёх downloader'ов — **не входит в текущие фазы**.

**Что нужно адаптировать в существующем коде (минимально):**
- `run.js` / `App.js` сегодня печатают прогресс через `cli-progress`. Для парсинга из Rust нужен **structured-progress mode** (например, JSON-строки в stdout по флагу `--json-progress` или env-переменной). Это единственное обязательное изменение downloader'а на MVP.

## Фазы реализации с критериями приёмки

Каждая фаза = независимо проверяемый кусок. Не двигаемся к N+1, пока приёмка по N не пройдена.

### F0 — Tauri scaffold + Bun sidecar handshake

**Делаем:** инициализируем Tauri project, добавляем Bun как sidecar в `tauri.conf.json`, фронт = одна кнопка «Test sidecar», Rust команда спавнит `bun -e 'console.log("hi from bun")'`, возвращает stdout во фронт.

**Приёмка:**
- `bun tauri dev` (или эквивалент) запускает окно на macOS.
- Клик по кнопке → в окне видно `hi from bun`.
- Sidecar-бинарь bundled (не PATH-зависимость).
- Никакой downloader-логики ещё не подключен.

### F1 — Один URL → скачивание через sidecar

**Делаем:** `Cmd+V` в окне берёт буфер → Rust спавнит `bun run.js <url>` (через sidecar) с фиксированным выводом в `~/Downloads`. Очереди и валидации пока нет. Окно показывает строку с URL и текстовым статусом (`running…` / `done` / `failed`).

**Приёмка:**
- Paste валидного Scribd URL → через X минут файл появляется в `~/Downloads`.
- При ошибке (например, отключить интернет) — статус становится `failed`, видна причина.
- Закрытие окна во время скачивания убивает child process (не остаётся zombie Bun).

### F2 — Очередь + список задач

**Делаем:** state machine очереди в Rust, фронт рендерит список (queued / downloading / done / failed). Paste во время активной задачи добавляет в `queued`, она стартует автоматически после завершения текущей.

**Приёмка:**
- Paste 3 URL подряд → видно 3 строки, первая `downloading`, две `queued`.
- По завершении первой автоматически стартует вторая.
- Все три завершаются `done`, файлы на диске.
- Paste одного и того же URL дважды — две строки (не дедуплицируем).

### F3 — Парсинг прогресса по страницам

**Делаем:** добавляем в `run.js` режим structured-progress (например, `--json-progress` или env `SCRIBD_DL_JSON_PROGRESS=1`), который вместо `cli-progress` пишет в stdout строки вроде `{"event":"progress","page":N,"total":M}`. Rust парсит, фронт показывает прогресс-бар в строке задачи.

**Приёмка:**
- Во время скачивания в строке виден `12 / 87` и прогресс-бар двигается монотонно к 100%.
- CLI без флага продолжает работать как раньше (батч-режим в терминале не сломан).
- `bun test` зелёный.

### F4 — Folder picker + персистенция

**Делаем:** в шапке окна — текущий путь вывода + кнопка «Change». Клик → нативный folder picker. Выбор сохраняется в Tauri app data store и подгружается при следующем запуске.

**Приёмка:**
- Дефолт = `~/Downloads` при первом запуске.
- Смена папки → следующее скачивание идёт туда.
- Закрыть и снова открыть приложение → выбранная папка та же.
- Изменение во время `downloading` задачи: уже идущая остаётся со старой папкой, новые `queued` уходят в новую.

### F5 — URL-валидация + alert на невалидный

**Делаем:** перед добавлением в очередь — regex против списка поддерживаемых доменов (scribd / slideshare / everand) из существующего роутера в `App.js`. Невалидный → нативный alert, в очередь ничего не добавляется.

**Приёмка:**
- Paste `https://example.com` → alert «Unsupported URL: scribd, slideshare, everand only», очередь не изменилась.
- Paste обычного текста (не URL) → тот же alert.
- Paste пустого буфера → тот же alert.
- Валидный URL — поведение F2 не сломалось.

### F6 — Коллизия имени файла → суффикс `(1)`

**Делаем:** перед записью PDF проверяем существование файла с целевым именем в папке вывода. Если занят — пробуем `Name (1).pdf`, `Name (2).pdf`, … до первого свободного. Логика — в downloader-слое (`src/utils/io/PdfGenerator.js` или вызывающем).

**Приёмка:**
- Скачать один и тот же URL дважды → второй файл = `Name (1).pdf`.
- Вручную положить в папку файл с именем, совпадающим с будущим скачиванием → новый сохранится как `... (1).pdf`.
- Существующий `bun test` зелёный, поведение CLI не изменилось (та же логика применяется и там — это не Tauri-only фича).

### F7 — Retry на failed-задачах

**Делаем:** в строке `failed` появляется кнопка **Retry**. Клик → задача возвращается в `queued` с тем же URL и параметрами; если очередь пуста и нет активной — сразу стартует.

**Приёмка:**
- Сымитировать фейл (отключить интернет / битый URL post-validation) → статус `failed`, кнопка Retry видна.
- Клик Retry → строка переходит в `queued` → стартует → доводится до `done`.
- Несколько failed-задач можно ретраить независимо.

### F8 — Quit guard

**Делаем:** перехват события закрытия окна / выхода из приложения. Если есть `downloading` или `queued` задачи → нативный confirm dialog «Идёт скачивание / задачи в очереди. Закрыть?». Cancel → ничего не происходит. OK → child process убивается, приложение закрывается.

**Приёмка:**
- Закрыть окно при пустой очереди → закрывается без диалога.
- Закрыть окно во время `downloading` → диалог. Cancel → окно остаётся, скачивание продолжается.
- Закрыть окно с `queued`-задачами без активной → диалог.
- OK → процесс sidecar убит, нет zombie.

### F9 — Packaging + smoke test на чистой macOS

**Делаем:** `tauri build` → `.app` + `.dmg`. Sidecar Bun + всё что нужно для Puppeteer (или сам Chromium) — внутри бандла. Проверяем на машине, где нет dev-окружения (нет Bun в PATH, нет Node).

**Приёмка:**
- DMG ставится на чистую macOS (или второй аккаунт без dev tools).
- Запуск из Launchpad → окно открывается.
- Paste валидного URL → файл скачался в `~/Downloads`.
- Размер DMG зафиксирован в README (для понимания, насколько Rust-rewrite того стоит позже).

## Зависимости и предположения

- **Bun 1.3.14** + текущий downloader продолжает работать как раньше — Tauri его не модифицирует, только параметризует флагом structured-progress.
- **Puppeteer + Chromium**: подход с sidecar означает, что Chromium либо качается при первом запуске (текущее поведение `puppeteer`), либо bundled. На F9 надо явно решить — но это упаковочный вопрос, не архитектурный.
- **Tauri 2.x** (актуальная мажорная на 2026-06-09). Все ссылки на sidecar-API и app-data store — из v2.
- **Последовательность скачиваний** — предположение, не явный пользовательский выбор. Если позже захочется параллелизма — это надстройка над F2, не блокер.

## Outstanding questions (решить в `/ce-plan` или позже)

- **Что показывает строка задачи до того, как известен заголовок документа?** Сейчас заголовок становится известен только в середине скачивания. Варианты: показывать URL, потом подменять на title; или показывать домен + первые N символов URL.
- **Code signing / notarization для macOS** — без них при первом запуске пользователь увидит Gatekeeper-предупреждение. Допустимо для self-use, но если планируется делиться билдом — нужно решить.
- **Размер DMG**: если sidecar Bun + Chromium через Puppeteer окажется > 300-500 МБ, это может ускорить решение про Rust-rewrite. Зафиксировать на F9 как реальный сигнал.
- **Concurrency**: сейчас зафиксировано «последовательно». Если на практике окажется, что Puppeteer-инстансы можно дёшево параллелить — обсудить отдельно, не блокер MVP.
- **Идемпотентность retry**: если задача упала на середине (часть страниц скачана), retry начинает с нуля — норма для MVP, но стоит зафиксировать.
